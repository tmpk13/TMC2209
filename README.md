# TMC2209
![https://www.lcsc.com/datasheet/C2150710.pdf](imgs/tmc2209.png)

Firmware for driving 3 TMC2209 stepper drivers over a shared half-duplex
UART, with STEP/DIR generated by PIO. Builds for both the **Raspberry Pi
Pico 2 (RP2350A)** and the original **Raspberry Pi Pico (RP2040)** from
the same source тАФ the chip is selected by a cargo feature, which also
swaps in the right linker memory map via `build.rs`. Pin assignments live
in [src/board.rs](src/board.rs) тАФ change them there and nothing else.

## Building

The default target is the Pico 2 (RP2350). Explicit aliases live in
`.cargo/config.toml` so you don't have to remember the triple + feature
combo.

```sh
# Pico 2 / RP2350 (default)
cargo build --release
cargo run   --release          # also generates firmware.uf2

# Pico 2 / RP2350, explicit
cargo build-pico2 --release
cargo run-pico2   --release

# Original Pico / RP2040
cargo build-pico --release
cargo run-pico   --release     # generates firmware-pico.uf2
```

The `run-*` aliases invoke `elf2uf2-rs` (the runner set in
`.cargo/config.toml`) to produce a UF2 next to the ELF. To flash, hold
BOOTSEL while plugging the Pico in and copy the UF2 onto the mounted
`RP2350` (or `RPI-RP2` for the original Pico) drive:

```sh
cp firmware.uf2 /media/$USER/RP2350/         # Pico 2
cp firmware-pico.uf2 /media/$USER/RPI-RP2/   # original Pico
```

After flashing, the device re-enumerates as a USB-CDC serial port
(`16c0:27dd`, product `tmc2209-fw`) тАФ usually `/dev/ttyACM0` on Linux.

If you have an SWD debug probe, uncomment the `runner = "probe-rs run тАж"`
line in `.cargo/config.toml` for the matching target and `cargo run` will
flash + stream defmt logs directly. Without a probe you only get UF2 mass
storage flashing and no defmt output.

### Target installation

The two targets are only set up once:

```sh
rustup target add thumbv8m.main-none-eabihf  # RP2350
rustup target add thumbv6m-none-eabi         # RP2040
cargo install elf2uf2-rs                     # UF2 packager (required)
```

## Host tools

Python tools live in [host/](host/) and use `pixi`. Run from the repo root:

```sh
pixi run tui [/dev/ttyACM0]   # Textual jog/config TUI
pixi run web [/dev/ttyACM0]   # FastHTML + p5.js browser UI at :8000
```

### Single-wire UART bus

All three TMC2209s share one PDN_UART line. The RP2350 speaks on GP8 (TX) and
listens on GP9 (RX) тАФ on a single-wire bus the two must meet:

```
  GP8  тФАтФАтФмтФАтФА 1k╬й тФАтФАтФмтФАтФАтЦ║ PDN_UART (J0, J1, J2, all tied together)
         тФВ         тФВ
  GP9  тФАтФАтФШ         тФВ
                   тФФтФАтФА (optional 10k╬й pull-up to VIO for idle-high)
```

The 1 k╬й in series with TX protects the RP2350 output driver when a TMC2209
is replying. The MCU always sees its own transmitted bytes echoed back first;
the firmware's `tmc2209::Reader` walks past the echo and surfaces only the
driver response.

Baud rate is 115200 (see `board::TMC_BAUD`).

**BIGTREETECH swap rx/tx**

### TMC2209 UART addresses

Each driver needs a unique address on the shared bus. Strap MS1/MS2 at power-up
**before VIO comes up** тАФ the TMC2209 latches the address once at boot:

| Joint | MS2 | MS1 | UART address |
|-------|-----|-----|--------------|
| J0    | GND | GND | 0            |
| J1    | GND | VIO | 1            |
| J2    | VIO | GND | 2            |

(A fourth address, `3 = MS2+MS1 both high`, is available if you ever add J3.)

After boot the MS1/MS2 pins are free to be reused as DIAG/INDEX тАФ but since the
firmware sets `GCONF.mstep_reg_select = 1`, microstep resolution comes from
CHOPCONF over UART, not from MS pins, so you can just leave them on the
address straps.

### Motor power

| TMC2209 pin | Connect to |
|-------------|------------|
| VM          | Stepper supply, 4.75 V тАУ 29 V (recommended 12тАУ24 V) |
| GND (power) | Supply return, shared with Pico 2 GND |
| 100 ┬╡F cap  | VM тЖФ GND, as close to each driver as possible |
| A1, A2      | Motor coil A |
| B1, B2      | Motor coil B |

The sense resistor on the breakout determines the current-scale mapping; the
firmware currently writes `IRUN = 16, IHOLD = 8` in
[src/driver.rs](src/driver.rs). Adjust those when you know your board's
`RSENSE` value and your motor's rated current.

### Host connection

USB-C / micro-USB straight into the Pico 2's native USB port. The firmware
enumerates as a CDC-ACM serial device (`16c0:27dd`, product string
`tmc2209-fw`).

## Scaling to 8 drivers

The TMC2209 only exposes 2 address bits (MS1/MS2), so a single shared
PDN_UART line tops out at 4 drivers. Eight drivers therefore need **two
UART buses**, each carrying up to 4 addressed drivers. Step generation
needs 8 PIO state machines, which fits cleanly into PIO0 + PIO1 (4 SMs
each) тАФ this layout works on Pico 2 (RP2350) and the original Pico
(RP2040) without using PIO2.

| GP1        | UART0 RX     | Bus B       | J4..J7 shared | тАФ |

| GP3        | STEP1        | PIO0 SM1    | J1    | 1 (Bus A)    |
| GP4        | STEP2        | PIO0 SM2    | J2    | 2 (Bus A)    |
| GP5        | STEP3        | PIO0 SM3    | J3    | 3 (Bus A)    |
| GP6        | DIR0         | тАФ           | J0    | тАФ            |
| GP7        | DIR1         | тАФ           | J1    | тАФ            |

### Two UART buses

Each bus is wired exactly like the 3-driver bus above тАФ TX and RX tied
together through a 1 kOhm series resistor on the MCU side, with the
joined node fanned out to the PDN_UART pin of every driver on that bus.
Optional 10 kOhm pull-up to VIO per bus.

```
Bus A:  GP8 --1k-- + -- PDN_UART (J0, J1, J2, J3)
        GP9 ------ +

Bus B:  GP0 --1k-- + -- PDN_UART (J4, J5, J6, J7)
        GP1 ------ +
```

### Address straps (8 drivers)

Address is per-bus, so J0 and J4 both strap to address 0 (just on
different wires):

| Joint | Bus | MS2 | MS1 | UART addr |
|-------|-----|-----|-----|-----------|
| J0    | A   | GND | GND | 0 |
| J1    | A   | GND | VIO | 1 |
| J2    | A   | VIO | GND | 2 |
| J3    | A   | VIO | VIO | 3 |
| J4    | B   | GND | GND | 0 |
| J5    | B   | GND | VIO | 1 |
| J6    | B   | VIO | GND | 2 |
| J7    | B   | VIO | VIO | 3 |

### Firmware changes required

This is a wiring map only тАФ the current firmware drives 3 joints on a
single bus (see [src/board.rs](src/board.rs) and
[src/driver.rs](src/driver.rs)). To actually run 8 drivers you need to:

- Add a second `DriverBus` for UART0 on GP0/GP1 and route J4..J7 to it.
- Extend `JointId` to `J0..J7` and split `addr()` so it returns the
  per-bus address (joint index mod 4).
- Bring up PIO1 alongside PIO0 in `motion.rs` and add 4 more STEP/DIR
  pin aliases in `board.rs`.
- Reassign EN to GP20 (the existing GP10 is reused as DIR2 in this
  layout).
